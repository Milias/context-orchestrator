# Design: Parallel Autonomous Agents via Dynamic Context Orchestration

**Date:** 2026-03-14
**Status:** Draft
**Prerequisites:** Research docs 05, 07, 20, 22, 24; Design doc 05 (git graph nodes)

---

## 1. Motivation

The context-orchestrator can create plans and tasks via the `plan` and `add_task` tools, producing `WorkItem` nodes in the graph. Today, these nodes are inert — no agent picks them up. The conversational agent creates tasks but cannot execute them in parallel. A user asking "implement JWT auth and add logging" must wait for the agent to do both sequentially in a single conversation turn.

The deeper problem: the system has a single, persistent "primary agent" with a continuous idle/wake loop. This architecture cannot scale to parallel work. When the agent creates a plan with 5 tasks, those tasks should be executed concurrently by independent agents, each with its own context window, its own git worktree for file isolation, and its own lifecycle.

**Design principle:** There are no long-running agents. There are selective context combinations generated by traversing the graph. Every graph event — a user message, a work item activation, a question — triggers the same pipeline: gather candidates → score → budget → render → spawn ephemeral agent → execute → terminate. The only difference between agent types is the **context policy**: how the graph is traversed and what nodes are selected.

This follows Gas Town's GUPP principle: "If there is work on your Hook, YOU MUST RUN IT." Agents don't wait for permission — they claim work atomically and proceed.

## 2. Architecture Overview

```
GraphEvent (MessageAdded { User }, WorkItemStatusChanged { Active }, ...)
  │
  ├─ Event Dispatch matches event to ContextPolicy
  │
  ├─ Phase A: Candidate Gathering (heuristic, O(V+E))
  ├─ Phase B: Deterministic Scoring (edge-weighted + recency)
  ├─ Phase C: [Configurable] LLM Refinement (meta-LLM re-ranks)
  ├─ Phase D: Budget Allocation (essential/important/supplementary)
  ├─ Phase E: Context Rendering (nodes → ChatMessage format)
  │
  ├─ ContextBuildingRequest node records selection in graph
  │
  ├─ [Task agents only] Create git worktree for file isolation
  │
  └─ Spawn ephemeral agent
       ├─ Inner tool loop: LLM → tools → persist → repeat
       ├─ Completion: propose Done with confidence level
       └─ Terminate
```

### 2.1 All Agents Are Ephemeral

No "primary agent" with a continuous loop. Every graph event spawns a fresh agent that runs one activation (bounded inner tool loop) and terminates. The outer idle/wake loop from the current `run_agent_loop()` is removed entirely.

A user message is just another graph event. `MessageAdded { User }` triggers a conversational agent with a simple branch-walk context. `WorkItemStatusChanged { Active }` triggers a task agent with a scored, scoped context. Both go through the same spawning pipeline.

### 2.2 Three-Layer Communication

Following the validated pattern from research docs 07 and 22:

| Layer | Purpose | Implementation |
|-------|---------|----------------|
| **Graph State** (durable) | Source of truth, queryable | `ConversationGraph` with `Arc<RwLock>` |
| **Event Log** (notifications) | Broadcast mutations | `EventBus` via `tokio::sync::broadcast` |
| **Ephemeral Signals** (coordination) | Tool results, phase tracking | `mpsc::UnboundedSender<TaskMessage>` |

Agents communicate exclusively through the graph (blackboard pattern). No direct agent-to-agent connections. Tool results are routed via the `AgentRegistry`'s per-agent channels.

## 3. ContextPolicy Enum

The core abstraction. Determines how context is built for an agent and how it interacts with the graph.

```rust
/// Determines how context is built for an agent.
pub enum ContextPolicy {
    /// Interactive chat on the main branch. Context = branch history.
    Conversational,
    /// Focused execution of a specific work item. Context = scored subgraph.
    TaskExecution { work_item_id: Uuid },
}
```

### 3.1 Policy Methods

| Method | Conversational | TaskExecution |
|--------|---------------|---------------|
| `build_context(graph, agent_id)` | Walk branch history | Score + budget from work item subgraph |
| `initial_parent(graph)` | `graph.active_leaf()` | Synthetic user message rooted at WorkItem |
| `record_message(graph, parent, node)` | `graph.add_message()` (updates branch) | `graph.add_reply()` (no branch update) |
| `tool_filter()` | `None` (all tools) | Whitelist: file tools + `update_work_item` + `ask` + `answer` |

No `is_wake_event()` method — agents are ephemeral and do not idle.

### 3.2 Extensibility

New policies are added as enum variants. The event dispatch maps events to policies:

```rust
match event {
    MessageAdded { User, .. } => spawn(Conversational, None),
    WorkItemStatusChanged { Active, .. } => spawn(TaskExecution { work_item_id }, Some(worktree)),
    // Future: QuestionAdded { Llm, .. } => spawn(QuestionResponse { question_id }, None),
}
```

## 4. Graph Model: RespondsTo Chains

Task agents do NOT use branches. Their messages form `RespondsTo` chains rooted at the WorkItem node.

```
WorkItem (task-123, "Implement JWT middleware")
  ← RespondsTo ─ SyntheticUserMsg ("Execute: Implement JWT middleware\n\nCreate token validation...")
    ← RespondsTo ─ AssistantMsg1 [tool_use: read_file, write_file]
      ← Invoked ─ ToolCall (read_file "src/auth.rs")
        ← Produced ─ ToolResult (file contents)
      ← Invoked ─ ToolCall (write_file "src/middleware/jwt.rs")
        ← Produced ─ ToolResult (success)
    ← RespondsTo ─ AssistantMsg2 [tool_use: update_work_item Done]
```

### 4.1 New Graph Method: `add_reply()`

Adds a node + `RespondsTo` edge WITHOUT updating any branch pointer. Identical to `add_message()` minus `self.branches.insert(...)`. The existing `add_message()` delegates to `add_reply()` + branch update.

### 4.2 Forward Index: `reply_children`

New runtime index `reply_children: HashMap<Uuid, Vec<Uuid>>` (serde-skipped, built at deserialization like `responds_to` and `invoked_by`). Enables O(depth) forward traversal from any root to its chain leaf via `find_chain_leaf()`.

## 5. ContextBuildingRequest — Context as a Graph Citizen

Every context construction operation is recorded as a graph node. This makes context building observable, auditable, and enables LLM-guided selection.

### 5.1 Node Definition

```rust
pub enum ContextBuildStatus {
    Requested,
    Building,
    Built,
    Consumed,
    FallbackUsed,
    Failed,
}

pub enum ContextTrigger {
    UserMessage,
    TaskExecution { work_item_id: Uuid },
    QuestionResponse { question_id: Uuid },
}

pub enum ContextPolicyKind {
    Conversational,
    TaskExecution,
}

Node::ContextBuildingRequest {
    id: Uuid,
    trigger: ContextTrigger,
    policy: ContextPolicyKind,
    status: ContextBuildStatus,
    candidates_count: u32,
    selected_count: u32,
    token_count: Option<u32>,
    agent_id: Uuid,
    created_at: DateTime<Utc>,
    built_at: Option<DateTime<Utc>>,
}
```

### 5.2 New Edge Types

| Edge | From | To | Meaning |
|------|------|----|---------|
| `SelectedFor` | ContextBuildingRequest | any node | "this node was included in the context window" |
| `ConsumedBy` | ContextBuildingRequest | Message (assistant) | "this context produced this response" |

Reuse existing `About` edge: ContextBuildingRequest → trigger anchor node.

### 5.3 New Events

- `ContextBuildingRequested { node_id, agent_id, trigger }`
- `ContextBuildingCompleted { node_id, selected_count, token_count }`
- `ContextConsumed { node_id, assistant_id }`

### 5.4 Lifecycle

The entire lifecycle happens within a single ephemeral agent execution:

```
Agent spawns
  → Create CBR (Requested)
  → Gather candidates, transition to Building
  → Score + [LLM refine] + budget + render
  → Record SelectedFor edges, transition to Built
  → Call LLM, stream response, persist
  → Add ConsumedBy edge, transition to Consumed
  → Agent terminates
```

### 5.5 Return Type

`extract_messages()` returns `ContextBuildResult` instead of a bare tuple:

```rust
pub struct ContextBuildResult {
    pub system_prompt: Option<String>,
    pub messages: Vec<ChatMessage>,
    pub selected_node_ids: Vec<Uuid>,
}
```

Both `conversational::build_messages()` and `task_execution::build_messages()` collect contributing node IDs as they traverse, at zero additional cost.

## 6. Context Scoring Pipeline

The core "context orchestration" capability. Replaces naive "include everything" with scored, budgeted, tiered selection.

### 6.1 Hybrid Layered Model

```
Phase A: Candidate Gathering ─── heuristic, O(V+E), always runs
Phase B: Deterministic Scoring ── edge-weighted + recency, zero API cost
Phase C: LLM Refinement ───────── configurable, meta-LLM re-ranks candidates
Phase D: Budget Allocation ────── token partitioning by tier
Phase E: Context Rendering ────── nodes → ChatMessage format
```

Config: `context_selection = "heuristic" | "llm_guided"` (default: `"heuristic"`).

### 6.2 Phase A: Candidate Gathering

Heuristic expansion from the trigger node, bounded at ~200 candidates:

1. Walk `RespondsTo` ancestors from branch leaf / chain leaf
2. All `WorkItem` nodes with status != Done
3. `Question`/`Answer` nodes relevant to this agent
4. Last N `ToolCall`/`ToolResult` pairs (configurable, default 5)
5. `GitFile` nodes (from doc 05 git watcher)
6. `GitCommit` summaries (recent, from doc 05)
7. `ApiError` nodes (for error recovery context)
8. `RelevantTo` edge fan-out from the trigger node

### 6.3 Phase B: Deterministic Scoring

**Edge-weighted scoring** (Strategy K from doc 22): score each candidate by shortest weighted distance from the trigger node.

| Edge Kind | Weight | Rationale |
|-----------|--------|-----------|
| `RespondsTo` | 1.0 | Direct conversation chain |
| `Invoked`/`Produced` | 0.9 | Tool execution chain |
| `SubtaskOf` | 0.8 | Plan hierarchy |
| `RelevantTo` | 0.7 | Explicit relevance |
| `Asks`/`Answers`/`About` | 0.6 | Q/A relationships |
| `Touches` | 0.5 | Git file relationships |
| All others | 0.3 | Weak signal |

**Recency boost**: `score *= 1.0 / (1.0 + age_minutes.ln())`.

**Tier assignment**:
- `score >= 0.7` → Essential (full content, always included)
- `score >= 0.4` → Important (full content if budget allows)
- `score >= 0.2` → Supplementary (summary in system prompt)
- Below 0.2 → excluded

This is O(V+E), deterministic, zero API cost, incrementally updatable.

### 6.4 Phase C: LLM Refinement Layer

Only runs when `context_selection = "llm_guided"`. Takes scored candidates from Phase B and refines via a meta-LLM call.

**Node summary format** (one-line per node):
```
[MSG:user] id=a1b2 age=2m score=0.9 | "Can you implement JWT auth..." (80 chars)
[TASK] id=g7h8 status=active score=0.7 | "Create token validation middleware"
[GIT] id=s9t0 status=modified score=0.5 | "src/auth.rs" (+42/-18)
```

**Meta-LLM call**:
- Model: configurable (`context_selector_model`), defaults to main model, can use Haiku
- No tools (reduces prompt overhead)
- Response: JSON with selected node IDs and tiers (essential/important/supplementary)
- Cost: ~$0.008/call (Haiku) to ~$0.028/call (Sonnet)

**Caching**: Cache key = `(agent_id, graph.mutation_version)`. Invalidated by: new user message, new tool result, WorkItem status change. Expected savings: ~50-70% fewer meta-LLM calls.

**Fallback**: If meta-LLM fails → use Phase B's deterministic scores. Record as `FallbackUsed` on the ContextBuildingRequest.

### 6.5 Phase D: Budget Allocation

Token budget partitioned by tier:
- Essential: 60% of `max_context_tokens`
- Important: 30%
- Supplementary: 10% (compact summaries in system prompt)

Nodes within each tier ordered by score. Lowest-scored truncated first when tier exceeds budget.

### 6.6 Phase E: Context Rendering

- Essential/Important: full `ChatMessage` entries (reuse `build_assistant_message_with_tools()`)
- Supplementary: compact summaries in a "## Reference Context" system prompt section
- Git context: `build_git_section()` filtered to selected git nodes only

## 7. File Isolation via Git Worktrees

Task agents operate in isolated git worktrees. Each agent gets a full checkout on its own branch.

### 7.1 Lifecycle

```
Agent spawn:  git worktree add .worktrees/task-{id} -b task-{id}
Agent works:  file tools scoped to worktree path
Completion:
  Verified/High  → worktree persists for review/merge
  Failed/cancel  → git worktree remove --force
```

### 7.2 Security

`validate_path()` and `validate_path_for_write()` accept `root_dir: &Path` instead of `std::env::current_dir()`. The worktree path becomes the containment root for all file operations. Path traversal (`../`) is still blocked.

### 7.3 Tool Filtering Enforcement

`spawn_tool_execution()` validates `arguments.tool_name()` against the policy's allowed tool set before executing. Non-whitelisted tools return an error result without execution. This prevents LLM hallucinations of non-whitelisted tools.

### 7.4 Integration with Doc 05 (Git Graph Nodes)

Task agent worktree branches appear as `GitBranch` nodes via `repo.branches()` during `scan_git_state()`. They `BelongsTo` the same `GitRepository` — NOT separate repos. Worktree-specific file status requires opening the worktree repo via `Repository::open(worktree_path)`.

### 7.5 Startup Recovery

`cleanup_stale_worktrees()` runs during `App::startup()`. Enumerates `git worktree list --porcelain`, identifies stale worktrees, removes them.

### 7.6 Retry Handling

Before creating a worktree, check if it already exists (previous agent failed). If so: `git worktree remove --force` + `git branch -D task-{id}`, then recreate.

## 8. Completion Confidence

Task agents propose completion with a confidence level that determines whether work auto-accepts or routes for review.

### 8.1 Confidence Levels

```rust
pub enum CompletionConfidence {
    Verified,   // Cross-referenced, auto-accept
    High,       // Thorough implementation, auto-accept
    Moderate,   // Uncertain aspects, review recommended
    Low,        // Partially done, review required
    Failed,     // Could not complete, review required
}
```

### 8.2 Review Routing

- `Verified` / `High`: transition WorkItem to Done immediately
- `Moderate` / `Low` / `Failed`: stay Active, create review Question via Q/A system with `requires_approval: true`

### 8.3 Review Cycle

When the user answers the review question (via `QuestionAnswered` event):
- **Accept**: transition WorkItem to Done
- **Reject**: keep WorkItem at Active, which fires `WorkItemStatusChanged { Active }`, spawning a fresh task agent

This does NOT require the original agent to still be alive — the review cycle is handled entirely by event dispatch.

## 9. Ephemeral Agent Lifecycle

### 9.1 Spawning

```
GraphEvent fires
  → event_dispatch matches event to ContextPolicy
  → Guard: active_count < max_concurrent_agents
  → [TaskExecution only]: claim WorkItem, create worktree
  → Register agent in AgentRegistry
  → spawn_agent_loop(config) — ephemeral
```

### 9.2 Execution

```
run_activation()
  → Create ContextBuildingRequest (Requested)
  → Gather candidates, score, [LLM refine], budget, render
  → Record SelectedFor edges (Built)
  → for _ in 0..max_tool_loop_iterations:
      Call LLM (stream response)
      Record assistant message + ConsumedBy edge
      If tool_use: dispatch tools, wait for results
      If end_turn: break
```

### 9.3 Termination

```
Agent sends AgentEvent::Finished
  → Release claims, requeue orphaned questions
  → Remove from AgentRegistry
  → [TaskExecution]: worktree persists for review
  → Prune old ContextBuildingRequest nodes (TTL)
```

### 9.4 What Changes from Current Code

| Current | New |
|---------|-----|
| `primary_agent_id: Option<Uuid>` | Removed. All agents equal. |
| Outer idle/wake loop in `run_agent_loop()` | Removed. One activation, then terminate. |
| `is_wake_event()` free function | Removed. No idle state. |
| `AgentEvent::Idle` / `GraphEvent::AgentIdle` | Removed. Dead code. |
| `event_rx` subscription in agent loop | Removed. Agents don't subscribe to events. |
| `spawn_primary_agent()` | Replaced by unified `spawn_agent(policy, working_dir)`. |
| `is_primary()` predicate in TUI | Removed. Identify agents by ContextPolicy variant. |

## 11. Red/Green Team Analysis

### 11.1 Green Team

1. **Follows existing patterns.** `ContextPolicy` is an enum like `ToolCallArguments`, `WorkItemKind`, `GraphEvent`. `ContextBuildingRequest` follows the `Question` node pattern (request with lifecycle). `add_reply()` decomposes `add_message()` cleanly.

2. **Ephemeral agents simplify the system.** Removing the outer idle/wake loop eliminates: `AgentEvent::Idle`, `GraphEvent::AgentIdle`, `is_wake_event()`, `event_rx` subscription, `primary_agent_id`. The agent loop goes from ~130 lines to ~50.

3. **Git worktrees provide real isolation.** No advisory locks, no conflict detection needed during execution. Merge conflicts are a review-time concern, not a runtime concern.

4. **Hybrid scoring model.** Deterministic scoring is always available (zero cost, deterministic). LLM refinement is optional and configurable. Fallback guarantees the system never degrades below heuristic behavior.

5. **CompletionConfidence plugs into existing Q/A.** `requires_approval`, `PendingApproval`, `QuestionAnswered` — all already work. No new review system needed.

### 11.2 Red Team

| # | Issue | Severity | Resolution |
|---|-------|----------|------------|
| 1 | Worktree files invisible to main repo scanner | CRITICAL | Per-worktree status scan in `scan_git_state()` |
| 2 | Worktree branch conflicts on retry | HIGH | Force-remove + recreate; startup recovery |
| 3 | `find_chain_leaf()` O(E) without forward index | HIGH | `reply_children` forward index |
| 4 | Tool filtering not enforced | HIGH | Validate tool name before execution |
| 5 | Review cycle: who transitions WorkItem? | HIGH | `QuestionAnswered` handler in event_dispatch |
| 6 | Merge conflicts across worktrees | HIGH | Deferred: sequential merge. Future: overlap detection |
| 7 | CBR node explosion (30 nodes/round) | MEDIUM | TTL-based pruning on agent completion |
| 8 | Graph serialization under read lock | MEDIUM | Acceptable at current scale; clone-and-serialize later |
| 9 | Spawn not atomic between events | MEDIUM | Synchronous `register()` before async spawn |
