# Claude Code Subscription Usage via LiteLLM OAuth2

> **Date:** 2026-03-13 | Research into using a Claude Code subscription (Pro/Max plan) to power this application through LiteLLM's OAuth2 support or direct token usage.

---

## 1. Executive Summary

This application can use Claude Code subscription credits (Pro, Max 5x, Max 20x) instead of a pay-per-token API key — and **no code changes are required**. The app already sends both `x-api-key` and `Authorization: Bearer` headers on every request (`src/llm/anthropic.rs:85-86`), and supports a configurable base URL (`src/config.rs:42-43`).

Two viable paths exist:

1. **Direct OAuth token** — Run `claude setup-token`, set the resulting token as `ANTHROPIC_AUTH_TOKEN`, point `ANTHROPIC_BASE_URL` at `https://api.anthropic.com`. Done. No proxy needed.
2. **LiteLLM proxy with OAuth** — Configure LiteLLM with `forward_client_headers_to_llm_api: true` so it passes the Bearer token through to Anthropic. Keeps proxy benefits (logging, rate limiting, multi-model routing).

Both paths use the same token type (`sk-ant-oat01-*`), valid for 1 year, generated by the Claude Code CLI.

---

## 2. Current Architecture

### Auth Flow

The app resolves credentials through a priority chain in `src/config.rs:72-79`:

```
ANTHROPIC_AUTH_TOKEN → ANTHROPIC_API_KEY → error
```

`ANTHROPIC_AUTH_TOKEN` takes precedence. The resolved value is used as both an API key header and a Bearer token (`src/llm/anthropic.rs:85-86`):

```rust
.header("x-api-key", &self.api_key)
.header("authorization", format!("Bearer {}", self.api_key))
```

This dual-header approach is what makes subscription usage possible without code changes — Anthropic's API accepts OAuth tokens via the `Authorization: Bearer` header, and the `x-api-key` header is ignored when a valid Bearer token is present.

### Current LiteLLM Setup

The existing `run.sh` connects through a corporate LiteLLM proxy:

```bash
export ANTHROPIC_BASE_URL=https://litellm-v2.trading.imc.intra/
export ANTHROPIC_AUTH_TOKEN=sk-12Zcd0gUrgzXaepi3KVdbQ
```

This token is a LiteLLM master key, not an Anthropic credential. The proxy handles actual Anthropic authentication.

## 4. Option A: Direct OAuth Token (No Proxy)

The simplest path. Use the Claude Code CLI to generate a long-lived OAuth token and pass it directly to Anthropic's API.

### Setup Steps

1. **Install Claude Code CLI** (if not already installed):
   ```bash
   npm install -g @anthropic-ai/claude-code
   ```

2. **Generate OAuth token**:
   ```bash
   claude setup-token
   ```
   This opens a browser for Anthropic account authentication, then outputs a token with prefix `sk-ant-oat01-*`. The token is valid for approximately 1 year.

3. **Configure the app**:
   ```bash
   export ANTHROPIC_BASE_URL=https://api.anthropic.com
   export ANTHROPIC_AUTH_TOKEN=sk-ant-oat01-<your-token>
   export ANTHROPIC_MODEL=claude-sonnet-4-6
   ```

4. **Run**:
   ```bash
   cargo run
   ```

### How It Works

The app sends both headers on every request. When an OAuth token (`sk-ant-oat01-*`) is present in the `Authorization: Bearer` header, Anthropic's API:
- Authenticates via the Bearer token
- Bills against the subscription plan associated with the account
- Ignores the `x-api-key` header (which contains the same value — not a valid API key format, but harmless)

### Token Details

| Property | Value |
|---|---|
| Prefix | `sk-ant-oat01-` |
| Lifetime | ~1 year |
| Auto-refresh | No (must run `claude setup-token` again) |
| Scope | Full API access for the authenticated account |
| Revocation | Via Anthropic account settings |

### Advantages

- Zero dependencies beyond the app itself
- No proxy latency
- No proxy failure modes
- Simplest possible configuration

### Disadvantages

- No request logging or rate limiting (beyond Anthropic's own)
- No multi-model routing
- Token shared across all usage surfaces (web chat, Claude Code CLI, this app)
- No centralized access control for team use

## 5. Option B: LiteLLM Proxy with OAuth

Keep the LiteLLM proxy for its operational benefits, but configure it to forward the OAuth token to Anthropic instead of using a separate API key.

### LiteLLM Configuration

```yaml
# litellm_config.yaml
model_list:
  - model_name: claude-sonnet-4-6
    litellm_params:
      model: anthropic/claude-sonnet-4-6-20250929
  - model_name: claude-opus-4-6
    litellm_params:
      model: anthropic/claude-opus-4-6

general_settings:
  forward_client_headers_to_llm_api: true
  master_key: os.environ/LITELLM_MASTER_KEY
```

The critical setting is `forward_client_headers_to_llm_api: true`. This tells LiteLLM to pass the client's `Authorization` header through to the upstream API, rather than stripping it and using its own credentials.

### App Configuration

```bash
export ANTHROPIC_BASE_URL=http://localhost:4000   # LiteLLM proxy
export ANTHROPIC_AUTH_TOKEN=sk-ant-oat01-<your-token>
export ANTHROPIC_MODEL=claude-sonnet-4-6
```

### Request Flow

```
App → POST /v1/messages
      Headers: x-api-key: sk-ant-oat01-...
               Authorization: Bearer sk-ant-oat01-...
    ↓
LiteLLM (forward_client_headers_to_llm_api: true)
    → POST api.anthropic.com/v1/messages
      Headers: Authorization: Bearer sk-ant-oat01-...
              (x-api-key may or may not be forwarded — see Known Issues)
    ↓
Anthropic API → authenticates via Bearer token → bills to subscription
```

### Endpoint Patterns

LiteLLM exposes two endpoint patterns for Anthropic models:

1. **Standard** — `POST http://localhost:4000/v1/messages`
   LiteLLM translates the request. Requires models to be listed in `model_list`.

2. **Passthrough** — `POST http://localhost:4000/anthropic/v1/messages`
   LiteLLM proxies the request without transformation. No model list needed. Enabled by default in recent LiteLLM versions.

The app currently uses the standard pattern (it posts to `{base_url}/v1/messages`), which works with both approaches.

### Advantages

- Request logging and observability
- Rate limiting and spend tracking
- Multi-model routing (use different models without changing app config)
- Can add fallback models or load balancing
- Centralized configuration for team use

### Disadvantages

- Additional infrastructure to maintain
- Proxy adds latency (typically 5-20ms)
- Known header-forwarding bugs in LiteLLM (see Section 8)
- More complex debugging when things go wrong

## 6. Option C: LiteLLM with Per-User OAuth (Multi-User)

For scenarios where multiple users each bring their own Claude subscription.

### How It Works

Each user generates their own `sk-ant-oat01-*` token via `claude setup-token`. The app sends the user's token in the `Authorization: Bearer` header. With `forward_client_headers_to_llm_api: true`, LiteLLM forwards each user's token to Anthropic, which bills the corresponding subscription.

### LiteLLM Configuration

Same as Option B. No per-user configuration needed in LiteLLM — authentication is fully delegated to Anthropic via the forwarded headers.

### Key Consideration

LiteLLM's `master_key` is used for LiteLLM's own authentication (who can use the proxy), while the forwarded `Authorization` header handles Anthropic authentication (whose subscription gets billed). These are independent auth layers.

```
User A (Max 20x) → LiteLLM (master_key auth) → Anthropic (User A's token → User A's subscription)
User B (Max 5x)  → LiteLLM (master_key auth) → Anthropic (User B's token → User B's subscription)
```

### Advantages

- Each user's subscription is billed independently
- Centralized logging across all users
- No shared credentials

### Disadvantages

- Every user must install Claude Code CLI and run `setup-token`
- Token management burden distributed across users
- If any user's subscription lapses, only their requests fail

## 8. Known Issues

### LiteLLM Header Stripping

LiteLLM's `clean_headers()` function has been reported to strip `Authorization` headers in certain configurations:

- **GitHub Issue #13380**: Headers not forwarded to upstream providers
- **GitHub Issue #19618**: `forward_client_headers_to_llm_api` not working with certain model configurations
- **GitHub Issue #22398**: Authorization header stripped during passthrough

**Workaround**: Use the passthrough endpoint (`/anthropic/v1/messages`) which bypasses LiteLLM's request transformation pipeline. Alternatively, pin a known-good LiteLLM version after testing.

### Token Expiry During Long Sessions

OAuth tokens from `claude setup-token` are valid for ~1 year but are not auto-refreshed. If a token expires mid-session:

- The app will receive a `401 Unauthorized` response
- The existing retry logic (`src/llm/retry.rs`) will retry, but all retries will fail with the same auth error
- The user must run `claude setup-token` again and restart the app

**Mitigation**: The 1-year lifetime makes this unlikely during normal use. For long-running deployments, set a calendar reminder to rotate tokens.

### Subscription Usage Sharing

The OAuth token shares the subscription's usage quota across all surfaces:
- Claude web interface (claude.ai)
- Claude Code CLI
- This application
- Any other tool using the same token

Heavy usage in one surface reduces availability in others. Max plan rate limits are per-account, not per-application.

### Anthropic's Position on Third-Party Tool Usage

Anthropic's Terms of Service permit using the API with personal tools. The OAuth token mechanism was explicitly designed to enable tools like Claude Code to authenticate without separate API keys. Using it in a custom application for personal use follows the same pattern. However:

- Redistributing tokens or building a multi-tenant service may violate ToS
- Commercial use beyond personal tooling should be reviewed against current terms
- The token is tied to a personal account and should not be shared

## 10. Integration with This App

### What Changes in `run.sh`

For direct usage (Option A), replace the current `run.sh`:

```bash
#!/usr/bin/env bash
export ANTHROPIC_BASE_URL=https://api.anthropic.com
export ANTHROPIC_AUTH_TOKEN=sk-ant-oat01-<your-token>
export ANTHROPIC_MODEL='claude-sonnet-4-6'

cargo run "$@"
```

For LiteLLM proxy usage (Option B), the only change is the token value:

```bash
#!/usr/bin/env bash
export ANTHROPIC_BASE_URL=http://localhost:4000
export ANTHROPIC_AUTH_TOKEN=sk-ant-oat01-<your-token>
export ANTHROPIC_MODEL='claude-sonnet-4-6'

cargo run "$@"
```

### Why No Code Changes

The app's auth implementation is already subscription-compatible:

1. **Dual headers** (`src/llm/anthropic.rs:85-86`): The app sends both `x-api-key` and `Authorization: Bearer` with the same value. Anthropic's API accepts OAuth tokens via Bearer, making the `x-api-key` header a no-op for subscription auth.

2. **Config priority** (`src/config.rs:72-79`): `ANTHROPIC_AUTH_TOKEN` takes priority over `ANTHROPIC_API_KEY`, and the variable name already implies OAuth/Bearer usage.

3. **Configurable base URL** (`src/config.rs:42-43`): `ANTHROPIC_BASE_URL` defaults to `https://api.anthropic.com` but can point anywhere — direct API, LiteLLM, or any compatible proxy.

## 11. Red/Green Team Audit

### Green Team (Verification)

> Claims verified against available documentation and source code.

**Token format**: `claude setup-token` produces tokens with `sk-ant-oat01-` prefix. This is the OAuth Access Token format used by Claude Code for subscription-based authentication. Confirmed via Claude Code documentation.

**LiteLLM config**: The `forward_client_headers_to_llm_api: true` setting is documented in LiteLLM's general settings. The YAML syntax in this document matches LiteLLM's configuration format.

**Code references verified**:
- `src/config.rs:40-80` — `AppConfig` struct and `api_key()` method ✓
- `src/llm/anthropic.rs:85-86` — Dual `x-api-key` + `Authorization: Bearer` headers ✓
- `src/config.rs:72-79` — Token priority chain (`auth_token` → `api_key` → error) ✓

**Pricing** (as of research date):
- Max 5x: $100/month (5x Pro usage limits)
- Max 20x: $200/month (20x Pro usage limits)
- API Sonnet: ~$3/$15 per million input/output tokens
- API Opus: ~$5/$25 per million input/output tokens

### Red Team (Challenges)

**Challenge 1: Is direct token usage actually supported?**
The `claude setup-token` command is designed for Claude Code CLI authentication. Using it in a custom application is an undocumented use case. Anthropic could:
- Restrict the token to specific User-Agent strings
- Rate-limit tokens used outside Claude Code
- Change the token scope without notice

*Assessment*: Low risk. The token authenticates against the standard Messages API endpoint. There is no evidence of User-Agent restrictions. However, this is not an officially supported integration path.

**Challenge 2: What happens when LiteLLM strips headers?**
The known header-stripping bugs are real and documented. If `forward_client_headers_to_llm_api` fails silently, requests would arrive at Anthropic without the Bearer token, resulting in `401 Unauthorized` errors.

*Assessment*: Medium risk for Option B. Mitigated by using the passthrough endpoint or testing header forwarding before production use. Option A avoids this entirely.

**Challenge 3: Subscription usage sharing**
Heavy usage in this app could degrade the Claude web experience or CLI performance due to shared rate limits.

*Assessment*: Real concern for Max 5x plans. Max 20x provides substantially more headroom. Users should monitor their usage dashboard.

**Challenge 4: Token stored in plain text**
The `run.sh` approach stores the token in a plain-text shell script. If this file is committed to git, the token is exposed.

*Assessment*: Valid concern. The token should be stored in a `.env` file (already in `.gitignore`) or a secrets manager, not in `run.sh`. The example `run.sh` in Section 10 shows the token inline for clarity, but production usage should use environment variables from a secure source.
